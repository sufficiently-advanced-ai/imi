# Concept: The World Model as IMI's Organizing Surface

*Status: Working concept — Proposed. Date: 2026-05-29.*
*Feeds: `docs/prd/decision-state-and-world-model-prd.md`.*

---

## The one-sentence version

IMI should present each account, team, or engagement as a **world model**: a
durable, human-reviewed **stable layer** (the constitution — what's been
decided and is still in force) sitting above an auto-maintained **current
layer** (the situation — what changed, what's drifting, what needs attention),
with a **governed review ritual** that promotes facts from current to stable.

This is not a new feature bolted onto the product. It is the shape the product
should take. The constitution view, the commitment-health view, the Andon Cord
alert, and the pre-meeting brief are all *views into a world model* — they stop
being four separate surfaces and become one object seen from different angles.

---

## Where this comes from

The pattern comes from lived practice: running an entire company on a world
model that lives in a hand-maintained markdown file with exactly this
structure:

- **Stable layer** (above the `---`): manually maintained ground truth — the
  company, infrastructure, product state, relationships, priorities. Reliable
  background context. Changes deliberately, only after human review.
- **Current layer** (below the `---`): owned by a cron job. "Recent Activity,"
  "What Needs Attention," "Observations." Auto-generated by diffing evidence
  (Gmail, Calendar, ClickUp, Drive, GitHub) against the stable layer. Ephemeral,
  regenerated continuously, never hand-edited.

And there is already a **promotion ritual**: a weekly review that pulls a
window of evidence, diffs it against what the stable layer claims, surfaces
contradictions / stale claims / missing entries *conversationally*, asks the
owner to confirm or correct, shows a clean diff, and only then writes the
approved changes — updating the review date in the header. It explicitly
watches for the "fabricated deadline" pattern: the model drifting to reflect
*intended* actions rather than *completed* ones.

The insight: **this is the product.** The thing that keeps a single operator's
world model accurate — a two-layer file plus a governed weekly diff-and-promote
ritual — is precisely the thing organizations need for their decisions. IMI
builds the team version of it.

## Why this is the right organizing concept (not just a nice metaphor)

### 1. It maps cleanly onto the architecture we already have

| World model concept | IMI implementation today |
|---------------------|--------------------------|
| Stable layer (constitution) | Git-backed markdown entities + decision records — already the portable source of truth (`entity_file_service.py`, `signals/*.json`) |
| Current layer (situation) | Signals + drift/contradiction detection computed over the graph |
| Promotion ritual | The human-in-the-loop approval gate from the May 8 architecture session (Signals vs DecisionRecords, declarative routing) |
| "What changed since T" | Temporal queries — **already implemented in Semantica** (`get_state_at`, `what_changed`, `graph_as_of`) |

The corpus/cache split an advisor argues for ("the KB owns institutional
truth, the harness owns local recall") is *already* our storage model: git
markdown is the corpus, Neo4j is the cache. The world model just makes that
split **visible to the customer** as two layers of one surface, instead of
leaving it as an implementation detail.

### 2. It resolves the "compass, not camera" problem structurally

One user's surveillance concern — "how do you not make this creepy?" — has a clean
answer in this frame. The stable layer is the *shared, editable, human-owned*
constitution: the team's agreed truth. The current layer only ever *surfaces*
discrepancies for human judgment; it never enforces. Drift shows up as "this
seems to have moved" in the current layer, and a human decides whether to
promote a correction into the stable layer. The architecture itself is the
compass: the machine observes and routes; the human decides what becomes truth.

### 3. It makes the demo honest *and* compelling

The review ritual's conversational diff is the single best demo we
have, and it already works on real data. The team version: "Here's your
constitution. Here's what changed across your meetings this week. Here are three
things that look like they drifted from what you decided. Confirm, correct, or
promote." That's "show, don't tell" — the only thing that has ever landed with
advisors — built from the customer's own transcripts.

### 4. It unifies the four product surfaces into one

- **Constitution view** = read the stable layer.
- **Commitment-health view** = the stable layer's decisions colored by current-layer status (active / stale / contradicted / superseded).
- **Andon Cord alert** = a current-layer event crossing a threshold (push).
- **Pre-meeting brief** = the slice of the world model relevant to an upcoming meeting/account.
- **Account onboarding** = "read the world model" for a new team member.

One object, five entry points. That's a far stronger product story than five
features.

## The model in detail

### Stable layer — the constitution

The durable, human-governed record. Holds anything that should survive the tool,
the employee, and the quarter:

- **Decisions** with rationale, owner, date, and lifecycle state.
- **Commitments** with owner and due context.
- **Standing rules / policies** the team operates under.
- **Entities and relationships** (people, accounts, projects) that give
  decisions their context.

Properties: human-reviewable, editable, exportable, diffable. Changes are
deliberate and attributable. **The stable layer wins** when it disagrees with
the current layer — until a human promotes a change. (This is the advisor's rule #5,
and it's the governance backbone.)

### Current layer — the situation

The machine-owned working view, regenerated continuously from incoming
transcripts and the temporal graph. Holds:

- **What changed** since the last review (new decisions, status changes).
- **What's drifting** — conversations that moved away from a standing decision.
- **What needs attention** — stale commitments, zombie (expired-temporary)
  decisions, cross-person contradictions.
- **Candidate promotions** — facts the system thinks belong in the constitution,
  staged for human confirmation.

Properties: ephemeral, disposable, never the source of truth. Always labeled as
*candidate* / *observed*, never asserted as institutional truth until promoted.

### The promotion ritual — the governed boundary

The mechanism that makes the whole thing trustworthy and non-creepy. Periodic
(or event-triggered): the system presents the diff between current and stable,
the human confirms / corrects / promotes, the system writes approved changes to
the stable layer and stamps the review. Directly productizes the
`world-model-review` skill. This is also where the advisor's rules #4 ("if the harness
learns something durable, promote it back into the KB") and #6 ("if a memory
can't be inspected, corrected, exported, or governed, don't treat it as
institutional truth") become product behavior.

## What this is NOT

- **Not a static wiki.** The current layer keeps it alive; the ritual keeps it
  honest. The advisor's failure mode — "a place where PDFs go to die" — is avoided
  precisely because the stable layer is small, reviewed, and current, not
  comprehensive and drifted.
- **Not auto-promotion.** Nothing enters the stable layer without human sign-off.
  Auto-promotion would reintroduce the "fabricated deadline" drift the review
  ritual exists to catch.
- **Not the graph.** The graph (Semantica over Neo4j) is the
  engine that computes the current layer and answers "what changed since T." The
  world model is the *surface*. Customers see the world model; the graph stays
  underneath.

## How it threads into the build (summary — see PRD for detail)

1. **Temporal substrate** (Semantica — already built): "what changed since T,"
   point-in-time state, and relationship evolution are implemented and tested
   (`get_state_at`, `get_active_relationships`, `TemporalQueryService`). This
   powers the current layer's "what changed" and the promotion diff. Remaining
   work is verification on real data and consistent validity-window coverage —
   not a new engine. (Graphiti/Kuzu are not needed.)
2. **Decision lifecycle + supersession**: gives stable-layer decisions real
   state (active / stale / superseded / temporary / zombie / conflicting) and
   lineage. Without this the constitution is just notes.
3. **Semantic contradiction detection**: upgrades the current keyword heuristic
   so the current layer can actually flag drift and cross-person conflict.
4. **World model export / view**: emits the two-layer world model as portable
   markdown (the constitution file + the situation digest) and renders it as the
   primary product surface. Closes the portability gap and *is* the constitution
   and commitment-health views.

A v0 of the world-model **view and export** can ship on top of today's data
before the temporal substrate is rich — it gets more powerful as layers 1–3 land.

## Open questions to pressure-test

- **Granularity.** One world model per account? Per engagement? Per team? The
  originating single-operator setup is per-company. A pilot client likely wants
  per-engagement (per client relationship). Probably configurable by domain template.
- **Review cadence.** The originating practice is weekly and manual. For a team, is it weekly,
  event-triggered (post-meeting), or both? The Andon Cord (push) and the review
  (pull) are two cadences of the same ritual.
- **Who promotes?** Single owner, or role-based (account lead promotes for their
  account)? Ties into the permissions framework's bucket model.
- **Stable-layer authorship.** Pure promotion-from-transcripts, or also a
  human-authored rules section (the `CLAUDE.md`-equivalent operating contract
  the advisor keeps by hand)? Likely both — extracted decisions *plus* deliberately
  written standing rules.
- **Conflict between two humans' promotions.** If two account leads disagree
  about what the constitution should say, the model needs a resolution path
  (this is the multi-person problem the advisor's solo vault never hits).
